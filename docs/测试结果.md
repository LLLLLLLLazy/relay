# Relay-BP Gong 输入测试结果

参考论文：

思路：

1. 用 CPU 跑 Relay-BP 行为模型，输入为 Gong 生成的 bivariate bicycle memory circuit
2. 对每个 code 分别测试 X/Z basis，并统计 `1000 shots` 的平均 BP 迭代次数
3. 假设 FPGA 上每次 BP iteration 为 `24 ns`，先换算整窗延迟
4. 按论文口径取 window size `W = d`，再除以 window rounds 得到每轮延迟

依照论文，令平均迭代次数为 \(\bar{I}\)，单次 BP iteration 延迟为\(t_{\mathrm{iter}} = 24\,\mathrm{ns}\)，窗口大小为 \(W=d\)，则

$$
T_{\mathrm{window}} = \bar{I} \cdot t_{\mathrm{iter}}
$$

$$
T_{\mathrm{round}}
= \frac{T_{\mathrm{window}}}{W}
= \frac{\bar{I} \cdot t_{\mathrm{iter}}}{d}
= \frac{\bar{I} \cdot 24\,\mathrm{ns}}{d}
$$

参数设置：

- 每个物理操作的错误率取 \(p=10^{-3}\)
- 滑动窗口长度取为对应码的距离，即 \(W=d\)
- FPGA 上一次 BP 迭代按 \(24\,\mathrm{ns}\) 估计
- 每个 code 的 X/Z basis 分别采样 1000 次，并统计平均迭代次数
- 解码器使用64整数实现，但是论文中有采用int4.2.4，也有正常64位
- 实验采用XZ Relay-BP-5，和论文中fig8保持一致

疑虑：

现在的 BP iteration 都固定为论文提供的 `24 ns`，但是论文中的 `24 ns`来自BB144在FPGA上实现

Tips：`ns/round` 取决于 \(\bar I / d\)，不是只看整窗延迟。BB72 的
window 较短，虽然 `ns/window` 很低，但只摊到 \(d=6\) 轮上，所以
`ns/round` 会比 BB90/BB108 略高。BB288 也类似，它的平均迭代次数相对
\(d=18\) 略高，导致摊到每轮后比相邻的 BB360 稍高一点。
